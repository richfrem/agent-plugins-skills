<technical_details>
**Core architectural pattern (Dashboard Pattern / Option 1.5):**
- `exploration-dashboard.md` lives at `exploration/exploration-dashboard.md` in the user's workspace
- `assets/templates/exploration-dashboard.md` is the canonical blank template the orchestrator scaffolds from
- State is read by checking `- [ ]` (unchecked) vs `- [x]` (checked) checkboxes â€” first unchecked = active phase
- Outcome file verification: for each `[x]` phase, orchestrator checks the listed Outcome file actually exists on disk

**The 6-block state machine in `exploration-workflow`:**
1. Bootstrap â€” check for dashboard, scaffold if missing, ask session name
2. Read State â€” find active phase, verify completed outcome files
3. Orientation â€” friendly status summary + soft confirmation (always shown)
4. Phase Routing â€” table: Phase 1â†’`discovery-planning`, Phase 2â†’`visual-companion`, Phase 3â†’`subagent-driven-prototyping`, Phase 4â†’`exploration-handoff`
5. HARD-GATE â€” wait for natural language affirmation before writing dashboard
6. Dashboard Write â€” check box, update Current Phase/Status, update Session Log, loop to Block 3

**Child skill pattern (all 4 child skills):**
- Dashboard Intercept at top: if `exploration-dashboard.md` exists â†’ redirect to `exploration-workflow`; if not â†’ proceed standalone
- Terminal Return at bottom: announce phase complete, invoke `exploration-workflow` if dashboard exists

**Scribe Phase collapse:**
- Old Phase 3 "Scribe" (`business-requirements-capture`, `user-story-capture`, `business-workflow-doc`) is now Stage 0 of `exploration-handoff`
- Stage 0 only runs if `exploration/prototype/` exists; `business-workflow-doc` only runs if Discovery Plan describes a process flow
- The 4-phase dashboard shows: Problem Framing â†’ Visual Blueprinting â†’ Prototyping â†’ Handoff & Specs

**Registry conflict resolved:**
- `skills/deferred/exploration-orchestrator/` deleted because the plugin registry dynamically scans for ANY `SKILL.md` file â€” a parked copy in deferred/ would cause conflicts

**Red-team findings (5 critical failure domains â€” NOT YET ADDRESSED):**
1. **State Machine Fragility**: NLP parsing of `- [x]` checkboxes is brittle â†’ fix: `.exploration-state.json` sidecar as true machine state, dashboard becomes read-only View
2. **HARD-GATE as prompt theatre**: Gate is enforced only by LLM prompts, not code â†’ fix: Python layer reads state.json and raises `GateViolationError` / `exit 1` if `discovery_plan_approved` is false
3. **Agent Bloat & Orchestration Overlap**: 10+ agents, poorly defined Skill vs Agent boundaries â†’ fix: extract routing into Python dispatch scripts, consolidate overlapping agents
4. **Ghost Agents and Silent Stub Failures**: `# Add your logic here` stubs return exit 0 silently â†’ fix: all stubs `exit 1` with "Not Implemented"; move `exploration-loop-orchestrator` to `deferred/`
5. **Dispatch Security**: `--dangerously-skip-permissions` used unconditionally; `ARG_MAX` truncation risk with large bash args â†’ fix: pipe context via stdin; restrict flag to Tier-1 sandbox with user consent
