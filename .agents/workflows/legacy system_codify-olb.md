# /codify-olb

**Purpose:** Generate comprehensive documentation for an Oracle Object Library by calling the reusable `/investigate-olb` analysis module, then filling the Overview template.

> 
> **Architecture:** This workflow is an **OUTPUT WORKFLOW**. It calls the shared `/investigate-olb` module for analysis, then focuses on documentation output.

**Template:** `plugins/legacy system/assets/templates/forms-sops/sop-codify-olb-template.md`
**Diagram:** `../skills/legacy-system-oracle-forms/references/diagrams/workflows/olb-discovery.mmd`

---

> **SYSTEM ENFORCEMENT PROTOCOL**
> 1. **MANDATORY EXECUTION**: You MUST invoke the `/legacy-system-oracle-forms_investigate-olb` skill via the Skill tool.
> 2. **PROHIBITED ACTION**: Manual XML reading, `grep`, or ad-hoc analysis is a VIOLATION.
> 3. **BLOCKING CONDITION**: If `temp/context-bundles/[OLBID]_context.md` is not generated by the skill, you MUST NOT proceed to documentation.
> 4. **FAILURE MODE**: If the tool fails, STOP and report the error. Do not fallback to manual work.

## Phase 1: Analysis (Call /investigate-olb)

### Step 2: Run Investigation Module
```bash
/legacy-system-oracle-forms_investigate-olb [OLBName]
```

**What this does:**
1. Initializes context (`.agent/skills/context-bundler/scripts/manifest_manager.py init --target [OLBName] --type olb`)
2. Locates OLB XML source
3. Runs the **Recursive Context Spiral** (`--direction downstream`, up to 3 levels)
4. Executes **OLB Mining** (SmartClasses, Visual Attributes, ObjectGroups, Triggers)
5. Performs **Dependency & Lineage Analysis** (`--direction both`: 1 level upstream, 3 levels downstream)

**Output:** Fully populated `temp/context-bundles/[OLBName]_context.md`

> **⚠️ DO NOT PROCEED until `/investigate-olb` completes successfully.**

### Step 3: Verify Context Completeness
```bash
view_file temp/context-bundles/[OLBName]_context.md
```
Confirm the bundle contains:
- [ ] Source XML
- [ ] SmartClass inventory
- [ ] Visual Attribute palette
- [ ] Consumer forms identified

## Phase 4: Feature Completion — Deterministic Closure Protocol (MANDATORY)

> **Every step below is MANDATORY.** Skipping any step is a protocol violation.
> The closure chain is: **Review → Accept → Retrospective → Merge → Verify → Intel Sync**

### Step 10: Accept feature
```bash
cd <PROJECT_ROOT>
spec-kitty accept --mode local --feature <SLUG>
```
> **Known Issue**: If accept fails with "missing shell_pid", use `--lenient` flag.

### Step 11: Retrospective (MANDATORY — not optional)
```bash
/spec-kitty_retrospective
```
**PROOF**: `kitty-specs/<SPEC-ID>/retrospective.md` must exist and be committed.

### Step 12: Pre-merge safety check
```bash
cd <PROJECT_ROOT>
git status
spec-kitty merge --feature <SLUG> --dry-run
```
Verify: in main repo root, clean status, no conflicts.

### Step 13: Merge from main repo
```bash
cd <PROJECT_ROOT>
spec-kitty merge --feature <SLUG>
```

> **LOCATION RULE**: ALWAYS run merge from the **main repository root**.
> NEVER `cd` into a worktree to merge.

### Step 14: Post-merge verification
```bash
git log --oneline -5
git worktree list
git branch
git status
```
Verify: merge visible, no orphaned worktrees, WP branches deleted, clean tree.
