# /legacy-system-oracle-forms_codify-form

**Purpose:** Generate comprehensive documentation for an Oracle Form by calling the reusable `/legacy-system-oracle-forms_investigate-form` analysis module, then filling the Overview template.

> 
> **Architecture:** This workflow is an **OUTPUT WORKFLOW**. It calls the shared `/legacy-system-oracle-forms_investigate-form` module for analysis, then focuses on documentation output.

**Policy:** `plugins/legacy system/references/rules/form_documentation_policy.md`

**Diagram:** `../skills/legacy-system-oracle-forms/references/diagrams/workflows/form-discovery.mmd`

> **SYSTEM ENFORCEMENT PROTOCOL**
> 1. **MANDATORY EXECUTION**: You MUST invoke the `/legacy-system-oracle-forms_investigate-form` skill via the Skill tool.
> 2. **PROHIBITED ACTION**: Manual XML reading, `grep`, or ad-hoc analysis is a VIOLATION.

## Phase 1: Analysis (Recursive Discovery)

### Step 2: Run Investigation Module
```bash
/legacy-system-oracle-forms_investigate-form [FormID]
```

**What this does:**
1.  **Initialize Context**: Loads RLM summaries and source code.
2.  **Recursive Context Spiral**: The agent engages as a **Recursive Language Model (RLM)**, pulling only required context incrementally through a 3-level spiral:
    *   **Level 1: UI Discovery (Downstream ONLY)**: Targets of `CALL_FORM` and `OPEN_FORM`.
    *   **Level 2: Processing Logic**: Attached Libraries (`.pll`) and shared logic.
    *   **Level 3: Data Access**: Database Packages, Procedures, and Views.
3.  **Multi-Pass Mining**: Extracts Blocks, Triggers, UI Layout, and Security.
4.  **Semantic Bundling**: Finalizes the high-fidelity "Ground Truth" bundle.

**Output:** Fully populated `temp/context-bundles/[FormID]_context.md`

> **⚠️ DO NOT PROCEED until `/investigate-form` completes successfully.**

### Step 3: Verify Context Completeness
```bash
view_file temp/context-bundles/[FormID]_context.md
```
Confirm the bundle contains:
- [ ] Source XML/Markdown
- [ ] All CALL_FORM targets
- [ ] Attached Libraries (PLL)
- [ ] Menu Module reference
- [ ] Key Database Packages
 
> **Exclude** recursively including other Forms/Reports/Menus to avoid Context Bloat.

### Step 4: Capture Validated Dependencies
```bash
/dependency-analysis_investigate-direct-dependencies [FormID]
```

*This generates: `temp/[FormID]_upstream.json`, `temp/[FormID]_downstream.json`, `temp/[FormID]_db_deps.json`*

## Phase 1.5: Business Rule Reconciliation (Mandatory)
> **Objective:** Ensure we verify existing rules before creating duplicates.

1.  **Mined Logic Check**: Review `temp/context-bundles/[FormID]_context.md` for mined Trigger logic.
2.  **Reconciliation Loop**:
    - For each complex validation or logic block found:
    - `/legacy-system-business-rules_investigate-business-rule "[Keyword]"`
    - **IF FOUND**: Use the existing `BR-XXXX` ID.
    - **IF NEW**: You **MUST** register it using the creation workflow:
      ```bash
      /legacy-system-business-rules_codify-business-rule --title "[Title]" --source "[FormID]" --priority P2
      ```
      > **CRITICAL**: Do NOT manually create BR files. Use the workflow to ensure proper ID generation and inventory sync.

## Phase 5: Feature Completion — Deterministic Closure Protocol (MANDATORY)

> **Every step below is MANDATORY.** Skipping any step is a protocol violation.
> The closure chain is: **Review → Accept → Retrospective → Merge → Verify → Intel Sync**

### Step 15: Accept feature
```bash
cd <PROJECT_ROOT>
spec-kitty accept --mode local --feature <SLUG>
```
> **Known Issue**: If accept fails with "missing shell_pid", use `--lenient` flag.

### Step 16: Retrospective (MANDATORY — not optional)
```bash
/spec-kitty_retrospective
```
**PROOF**: `kitty-specs/<SPEC-ID>/retrospective.md` must exist and be committed.

The retrospective includes the **Boy Scout Rule**: identify one improvement action:
- [ ] **Fix Code**: Repair a bug found during execution
- [ ] **Fix Docs**: Clarify a confusing workflow step
- [ ] **Log Task**: Create backlog item if too large

### Step 17: Pre-merge safety check
```bash
cd <PROJECT_ROOT>
git status
git worktree list
spec-kitty merge --feature <SLUG> --dry-run
```
Verify: in main repo root, clean status, no conflicts.

### Step 18: Merge from main repo
```bash
cd <PROJECT_ROOT>
spec-kitty merge --feature <SLUG>
```

> **LOCATION RULE**: ALWAYS run merge from the **main repository root**.
> NEVER `cd` into a worktree to merge. The `@require_main_repo` decorator blocks this.

If merge fails mid-way:
```bash
spec-kitty merge --feature <SLUG> --resume
```

### Step 19: Post-merge verification
```bash
git log --oneline -5
git worktree list
git branch
git status
```
Verify:
- [ ] Merge commit(s) visible in log
- [ ] No orphaned worktrees remain
- [ ] WP branches deleted
- [ ] Working tree is clean
