# /codify-db-package

**Purpose:** Complete documentation workflow for Oracle Database Packages. Delegates analysis to `/investigate-db-package`, then fills template and runs post-processing.


---

> **SYSTEM ENFORCEMENT PROTOCOL**
> 1. **MANDATORY EXECUTION**: You MUST invoke the `/legacy-system-database_investigate-db-package` skill via the Skill tool.
> 2. **PROHIBITED ACTION**: Manual XML reading, `grep`, or ad-hoc analysis is a VIOLATION.
> 3. **BLOCKING CONDITION**: If the investigation bundle is not generated by the skill, you MUST NOT proceed to documentation.
> 4. **FAILURE MODE**: If the tool fails, STOP and report the error. Do not fallback to manual work.

## Phase 0: Pre-Flight (The Gate)

```bash
/agent-orchestrator_plan "codify-db-package-[PackageName]"
```

**What this does:**
1.  **Context Check**: Verifies git branch and environment.
2.  **SDD Lifecycle**: Executes `/spec-kitty.specify` -> `/spec-kitty.plan` -> `/spec-kitty.tasks`.

> **⚠️ STOP:** Do not proceed to Phase 1 until the Orchestrator confirms the plan and tasks are generated.

## Phase 0.5: Create Worktree (MANDATORY for Clean Merge)

> **⚠️ CRITICAL**: This step creates the isolated workspace that enables `spec-kitty merge` to work properly.

### Step 1a: Create Implementation Worktree
```bash
spec-kitty implement WP01
```

**What this does:**
1. Creates `.worktrees/###-codify-db-package-[PackageName]-WP01/` directory
2. Checks out a feature branch for this work package
3. Sets up isolated workspace for implementation

**After running:**
- You are now in the worktree directory
- All implementation happens here (Phases 1-4)
- When done, `spec-kitty merge` will work correctly

## Phase 1: Analysis (Call /investigate-db-package)

### Step 2: Run Investigation Module
```bash
/legacy-system-database_investigate-db-package [PackageName]
```

**What this does:**
1. Initializes context (`.agent/skills/context-bundler/scripts/manifest_manager.py init --target [PackageName] --type package`)
2. Runs the **Recursive Context Spiral** (`--direction downstream`, up to 3 levels)
3. Executes **Package Mining** (API, Tables, Validation, Security)
4. Performs **Dependency & Lineage Analysis** (`--direction both`: 1 level upstream, 3 levels downstream)

**Output:** Fully populated `temp/context-bundles/[PackageName]_context.md`

> **⚠️ DO NOT PROCEED until `/legacy-system-database_investigate-db-package` completes successfully.**

### Step 3: Verify Context Completeness
```bash
view_file temp/context-bundles/[PackageName]_context.md
```
Confirm the bundle contains:
- [ ] Source Code (SQL spec and body)
- [ ] Public API inventory
- [ ] Downstream dependencies
- [ ] Upstream callers

## Phase 4: Feature Completion — Deterministic Closure Protocol (MANDATORY)

> **Every step below is MANDATORY.** Skipping any step is a protocol violation.
> The closure chain is: **Review → Accept → Retrospective → Merge → Verify → Intel Sync**

### Step 10: Accept feature
```bash
cd <PROJECT_ROOT>
spec-kitty accept --mode local --feature <SLUG>
```
> **Known Issue**: If accept fails with "missing shell_pid", use `--lenient` flag.

### Step 11: Retrospective (MANDATORY — not optional)
```bash
/spec-kitty_retrospective
```
**PROOF**: `kitty-specs/<SPEC-ID>/retrospective.md` must exist and be committed.

### Step 12: Pre-merge safety check
```bash
cd <PROJECT_ROOT>
git status
spec-kitty merge --feature <SLUG> --dry-run
```
Verify: in main repo root, clean status, no conflicts.

### Step 13: Merge from main repo
```bash
cd <PROJECT_ROOT>
spec-kitty merge --feature <SLUG>
```

> **LOCATION RULE**: ALWAYS run merge from the **main repository root**.
> NEVER `cd` into a worktree to merge.

### Step 14: Post-merge verification
```bash
git log --oneline -5
git worktree list
git branch
git status
```
Verify: merge visible, no orphaned worktrees, WP branches deleted, clean tree.
