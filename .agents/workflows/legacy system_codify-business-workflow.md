**When to use:**
This workflow handles the formal creation of a Business Workflow (BW) artifact. It is used in two primary scenarios:

### Scenario A: Embedded Sub-Workflow (Analysis)
Invoked by higher-level workflows like `/codify-form`, `/codify-library`, or `/codify-db-trigger`.
- **Trigger:** You find a multi-step process (e.g., "Case Intake Flow") while analyzing a specific file.
- **Goal:** Create a workflow to document the end-to-end process.
- **Source:** The Artifact ID (e.g., `JCSE0001`, `JUSTIN_INTAKE_PKG`).

### Scenario B: Human Intelligence (Standalone/Manual)
Invoked manually by a User providing context outside of code analysis (e.g., Jira Tickets, Policy Docs, User Stories).
- **Trigger:** "Hey, there is a report request process..." or [RCCJAM-832].
- **Goal:** Codify tribal knowledge or specific requirements.
- **Source:** "UserRequest", "Jira:[TicketID]", or "Manual".

### Scenario C: Augmenting Existing Workflows
You find an existing workflow (`BW-XXXX`) that partially matches but lacks detail.
- **Goal:** Update the summary or add new source links without creating a duplicate.
- **Action:** Use the Update command.

> **Note on Scenario D (Duplicates):**
> Any of the above scenarios (A, B, or C) may reveal that the logic is duplicated across the system.
> If **Investigation (Phase 1)** reveals duplicates, this workflow shifts to **Outcome D: Consolidation**.

**Policy:** `plugins/legacy system/references/rules/business_workflow_documentation_policy.md`

## Phase 1: Discovery & Investigation
**Critical Dependency:** This phase uses the recursive topic investigation.

### Step 1: Execute Investigation
Run the recursive investigation to determine if the workflow exists and gather context.

**Command:**
```bash
/legacy-system-business-workflows_investigate-business-workflow "[Topic Keyword]"
```

### Step 1b: Expand Context (The "Impact Radius" Rule)
If the initial investigation reveals **Consumer Forms/Reports** (e.g. usage lists in descriptions), you **MUST** add them to the context bundle.

> [!IMPORTANT]
> **Manifest Path:** The `--investigate` command creates a topic-specific manifest at:
> `temp/manifests/topic_[slug]_manifest.json`
> You **MUST** use `--manifest [PATH]` on all subsequent `manifest add` commands to target the correct context.

1.  **Capture Manifest Path:** Note the manifest path from the `--investigate` output (e.g., `temp/manifests/topic_case-intake_manifest.json`).
2.  **Identify Participants:** List all Forms/Reports that participate in the workflow.
3.  **Add to Bundle:** 
    ```bash
    /context-bundler_add --path [FILE] --note "[Why]" --manifest temp/manifests/topic_[slug]_manifest.json
    ```
4.  **Re-Bundle:** 
    ```bash
    /context-bundler_bundle --manifest temp/manifests/topic_[slug]_manifest.json
    ```
5.  **Analyze:** Review the updated bundle for workflow logic (e.g. state transitions, form navigation).
6.  **Recursion:** Repeat up to **3 Levels** if participants call other dependencies.
    *   *Constraint:* Analyze ONLY logic related to the workflow. Do not document unrelated form features.

### Step 2: Act on Investigation Outcomes
Review the "Key Outcomes" from the investigation:

**Outcome A: Workflow Exists (Match Found)**
- **Action:** Proceed to **Phase 2b: Update**.
- **Context:** Use the `temp/context-bundles/` context to augment the existing workflow.

**Outcome B: New Workflow Needed (Logic Found)**
- **Action:** Proceed to **Phase 2: Creation**.
- **Context:** Use the `temp/context-bundles/` context to populate the new workflow.

**Outcome C: No Logic Found**
- **Action:** STOP. Request human clarification.

**Outcome D: Duplicates Identified (Merge Required)**
- **Trigger:** Investigation finds multiple existing workflows describing the same process.
- **Action:** Create a task in `tasks/backlog/` titled `consolidate-workflows-[Topic].md`.
- **Note:** Do not attempt to merge them in this session. Flag them for future cleanup.

## Phase 2: Creation

### Step 3: Register Workflow & Create File
If no match found, register the workflow.
*If this is a Standalone workflow (no Form source), use "Manual" or "UserRequest" as SourceID.*

```bash
/legacy-system-business-workflows_investigate-business-workflow --register --title "[Title]" --source "[SourceID]" --priority [P1/P2/P3]
```
*(This command handles ID generation, duplicate checking, and template creation automatically)*

*Output:* `legacy-system/business-workflows/BW-NNNN-[Slug].md`

### Step 4: Fill Content (Safely)
Edit the new file using `plugins/legacy system/assets/templates/general-sops/sop-codify-business-workflow-template.md`.

> [!CAUTION]
> **Preserve Existing Analysis:** If this file already exists (e.g. from a previous analysis), **DO NOT OVERWRITE** specific findings or SME notes.
> **Augment Only:** Add your new findings to the appropriate sections.

**Required Context:**
- **Visual Check:** Ensure the ID matches the filename.
- **Trigger:** What starts this workflow?
- **Actors:** Who is involved? (Roles, Systems)
- **Sequence Diagram:** Mermaid diagram required.
- **Steps:** Ordered list of actions.

### Step 4b: Map Impact (Participants)
Identify and Document components that *participate* in this workflow.
- **Search:** Query usages of the main process trigger.
- **Analyze:** Check *how* forms participate (Entry Point vs Mid-Flow vs Exit).
- **Table:** Add "Participating Components" table to the BW file.
