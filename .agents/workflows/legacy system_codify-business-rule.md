**When to use:**
This workflow handles the formal creation of a Business Rule (BR) artifact. It is used in two primary scenarios:

### Scenario A: Embedded Sub-Workflow (Analysis)
Invoked by higher-level workflows like `/codify-form`, `/codify-library`, or `/codify-db-trigger`.
- **Trigger:** You find complex logic (e.g., "Audit Entry on Tab Switch") while analyzing a specific file.
- **Goal:** Create a rule to link back to the artifact.
- **Source:** The Artifact ID (e.g., `JCSE0030`, `QC_PKG`).

### Scenario B: Human Intelligence (Standalone/Manual)
Invoked manually by a User providing context outside of code analysis (e.g., Jira Tickets, Policy Docs, Court Inquiry scenarios).
- **Trigger:** "Hey, there is a logging rule for criminal case details..." or [RCCJAM-832].
- **Goal:** Codify tribal knowledge or specific requirements.
- **Source:** "UserRequest", "Jira:[TicketID]", or "Manual".

### Scenario C: Augmenting Existing Rules
You find an existing rule (`BR-XXXX`) that partially matches but lacks detail.
- **Goal:** Update the summary or add new source links without creating a duplicate.
- **Action:** Use the Update command.

> **Note on Scenario D (Duplicates):**
> Any of the above scenarios (A, B, or C) may reveal that the logic is duplicated across the system.
> If **Investigation (Phase 1)** reveals duplicates, this workflow shifts to **Outcome D: Consolidation**.

**Policy:** `plugins/legacy system/references/rules/business_rule_documentation_policy.md`

## Phase 1: Discovery & Investigation
**Critical Dependency:** This phase is now handled by a specialized workflow.

### Step 1: Execute Investigation
Run the recursive investigation workflow to determine if the rule exists and gather context.

**Command:**
```bash
/legacy-system-business-rules_investigate-business-rule "[Topic Keyword]"
```

### Step 1b: Expand Context (The "Impact Radius" Rule)
If the initial investigation reveals **Consumer Forms/Reports** (e.g. usage lists in descriptions), you **MUST** add them to the context bundle.

> [!IMPORTANT]
> **Manifest Path:** The `--investigate` command creates a topic-specific manifest at:
> `temp/manifests/topic_[slug]_manifest.json`
> You **MUST** use `--manifest [PATH]` on all subsequent `manifest add` commands to target the correct context.

1.  **Capture Manifest Path:** Note the manifest path from the `--investigate` output (e.g., `temp/manifests/topic_court-access-levels_manifest.json`).
2.  **Identify Consumers:** List all Forms/Reports that use the rule.
3.  **Add to Bundle:** 
    ```bash
    /context-bundler_add --path [FILE] --note "[Why]" --manifest temp/manifests/topic_[slug]_manifest.json
    ```
4.  **Re-Bundle:** 
    ```bash
    /context-bundler_bundle --manifest temp/manifests/topic_[slug]_manifest.json
    ```
5.  **Analyze:** Review the updated bundle for usage logic (e.g. `POST-QUERY` filtering).
6.  **Recursion:** Repeat up to **3 Levels** if consumers call other dependencies.
    *   *Constraint:* Analyze ONLY logic related to the business rule. Do not document unrelated form features.

### Step 2: Act on Investigation Outcomes
Review the "Key Outcomes" from the investigation:

**Outcome A: Rule Exists (Match Found)**
- **Action:** Proceed to **Phase 2b: Update**.
- **Context:** Use the `temp/context-bundles/` context to augment the existing rule.

**Outcome B: New Rule Needed (Logic Found)**
- **Action:** Proceed to **Phase 2: Creation**.
- **Context:** Use the `temp/context-bundles/` context to populate the new rule.

**Outcome C: No Logic Found**
- **Action:** STOP. Request human clarification.

**Outcome D: Duplicates Identified (Merge Required)**
- **Trigger:** Investigation finds multiple existing rules describing the same logic.
- **Action:** Create a task in `tasks/backlog/` titled `consolidate-rules-[Topic].md`.
- **Note:** Do not attempt to merge them in this session. Flag them for future cleanup.

## Phase 2: Creation

### Step 3: Register Rule & Create File
If no match found, register the rule.
*If this is a Standalone rule (no Form source), use "Manual" or "UserRequest" as SourceID.*

```bash
/legacy-system-business-rules_investigate-business-rule --register --title "[Title]" --source "[SourceID]" --priority [P1/P2/P3]
```
*(This command handles ID generation, duplicate checking, and template creation automically)*

*Output:* `legacy-system/business-rules/BR-NNNN-[Slug].md`

### Step 4: Fill Content (Safely)
Edit the new file.

> [!CAUTION]
> **Preserve Existing Analysis:** If this file already exists (e.g. from a previous analysis), **DO NOT OVERWRITE** specific findings or SME notes.
> **Augment Only:** Add your new findings to the appropriate sections.

**Required Context:**
- **Visual Check:** Ensure the ID matches the filename.
- **Logic:** Separate Logic (Pseudocode) from Implementation (PL/SQL).
- **Source Context:** Link back to the specific artifact (e.g., `Form: [JCSE0030]`, `Trigger: [PRE-INSERT]`) or cite "SME Input" directly.
- **Impact:** Identify Known Consumers (Forms/Reports) that rely on this rule.
- **Precision:** Analyze consumers (max depth 3) for rule-specific logic only.

### Step 4b: Map Impact (Consumers)
Identify and Document components that *use* this rule.
- **Search:** Query usages of the main logic function (e.g. `set_access_level`).
- **Analyze:** Check *how* they use it (Enforcement vs Display).
- **Table:** Add "Affected Components" table to the BR file.
